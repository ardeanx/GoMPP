GoMPP uses **PostgreSQL 16** with the **pgx/v5** native driver and connection pooling via `pgxpool`. Migrations are managed by [Goose](https://github.com/pressly/goose) and embedded in the binary.

---

## Connection Pool

The pool is configured via environment variables:

| Parameter           | Default | Description                      |
| ------------------- | ------- | -------------------------------- |
| `MaxConns`          | 25      | Maximum total connections        |
| `MinConns`          | 5       | Minimum idle connections         |
| `MaxConnLifetime`   | 60 min  | Maximum connection lifetime      |
| `MaxConnIdleTime`   | 30 min  | Maximum idle connection time     |
| `HealthCheckPeriod` | 1 min   | Background health check interval |

All queries use parameterised placeholders (`$1`, `$2`, ...) to prevent SQL injection.

## Table Reference

### users

| Column                | Type         | Constraints                                     | Description             |
| --------------------- | ------------ | ----------------------------------------------- | ----------------------- |
| `id`                  | UUID         | PK, default `gen_random_uuid()`                 | User ID                 |
| `email`               | VARCHAR(255) | NOT NULL, UNIQUE                                | Email address           |
| `username`            | VARCHAR(100) | NOT NULL, UNIQUE                                | Display name            |
| `password_hash`       | TEXT         | NOT NULL                                        | bcrypt hash             |
| `role`                | VARCHAR(20)  | NOT NULL, CHECK `(admin, user)`, default `user` | User role               |
| `is_active`           | BOOLEAN      | NOT NULL, default `true`                        | Account active status   |
| `avatar_url`          | TEXT         | —                                               | Avatar image path       |
| `google_id`           | TEXT         | UNIQUE                                          | Google OAuth subject ID |
| `last_login_at`       | TIMESTAMPTZ  | —                                               | Last login timestamp    |
| `password_changed_at` | TIMESTAMPTZ  | —                                               | Last password change    |
| `created_at`          | TIMESTAMPTZ  | NOT NULL, default `NOW()`                       | —                       |
| `updated_at`          | TIMESTAMPTZ  | NOT NULL, auto-updated                          | —                       |

**Indexes:** `email`, `username`, `role`

<Callout type="info">
  The `role` check constraint allows `admin` and `user`. The `super_admin` and
  `staff` roles are handled at the application layer.

## Automatic Triggers

All tables with `updated_at` columns use a shared trigger function:

```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

This ensures `updated_at` is always current without application-level handling.

## Query Patterns

### Atomic Job Claiming

Workers use `SELECT FOR UPDATE SKIP LOCKED` to atomically claim the next pending job:

```sql
SELECT * FROM transcode_jobs
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

This prevents race conditions between concurrent workers.

### Soft Deletes

Videos use a `deleted_at` column instead of physical deletion:

```sql
-- All video queries include:
WHERE deleted_at IS NULL
```

A partial index on `deleted_at WHERE deleted_at IS NULL` optimises these queries.

### UPSERT for Settings

Settings use PostgreSQL's `ON CONFLICT` for upsert operations:

```sql
INSERT INTO system_settings (key, value)
VALUES ($1, $2)
ON CONFLICT (key) DO UPDATE SET value = $2;
```
