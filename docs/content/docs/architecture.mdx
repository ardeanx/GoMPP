## Backend Architecture

The Go backend follows a **layered architecture** with strict dependency direction:

```
Handler (HTTP) вҶ’ Service (Business Logic) вҶ’ Repository (Data Access) вҶ’ Database
                                           вҶ’ Storage (File I/O)
```

### Layer Responsibilities

| Layer          | Package               | Role                                                          |
| -------------- | --------------------- | ------------------------------------------------------------- |
| **Router**     | `internal/router`     | Route definitions, middleware composition, CORS               |
| **Middleware** | `internal/middleware` | Auth, rate limiting, logging, RBAC, security headers, metrics |
| **Handler**    | `internal/handler`    | HTTP request/response handling, input validation              |
| **Service**    | `internal/service`    | Business logic, orchestration, external integrations          |
| **Repository** | `internal/repository` | SQL queries, data mapping, pagination                         |
| **Model**      | `internal/model`      | Structs, constants, request/response types                    |
| **Storage**    | `internal/storage`    | File operations abstracted behind `Backend` interface         |
| **Transcoder** | `internal/transcoder` | FFmpeg execution, worker pool, progress tracking              |
| **Config**     | `internal/config`     | Viper-based configuration from env vars                       |

### Key Design Decisions

**No ORM.** GoMPP uses raw SQL with `pgx/v5` for maximum control over queries, transactions, and connection pooling. The repository layer handles all SQL and data mapping.

**Interface-based storage.** The `storage.Backend` interface abstracts file operations, allowing seamless switching between local filesystem and S3-compatible storage without changing business logic.

**Stateless authentication.** JWT tokens with no server-side session storage. WebAuthn session state is signed with HMAC and embedded in the response rather than stored server-side.

**Atomic job claiming.** Transcoding jobs use PostgreSQL's `SELECT ... FOR UPDATE SKIP LOCKED` to enable safe concurrent processing by multiple workers without double-processing.

## Transcoding Pipeline

The transcoding system uses a producer-consumer pattern with a configurable worker pool:

```
Upload в”Җв”Җв–¶ Create Jobs в”Җв”Җв–¶ Worker Pool в”Җв”Җв–¶ FFmpeg в”Җв”Җв–¶ Storage
                               в”Ӯ
                          в”Ңв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”җ
                          в”Ӯ Worker 1в”Ӯв”Җв”Җв–¶ Probe вҶ’ Transcode вҶ’ Upload
                          в”Ӯ Worker 2в”Ӯв”Җв”Җв–¶ Probe вҶ’ Transcode вҶ’ Upload
                          в”Ӯ Worker Nв”Ӯв”Җв”Җв–¶ Probe вҶ’ Transcode вҶ’ Upload
                          в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### Job Lifecycle

```
pending вҶ’ processing вҶ’ completed
                    вҶҳ failed (auto-retry if attempts < max_attempts)
                         вҶҳ pending (retry)
```

1. **Upload** вҖ” Video file stored to `source/{videoID}/{filename}`
2. **Job Creation** вҖ” One `transcode_job` per default encoding preset
3. **Claiming** вҖ” Workers atomically claim pending jobs via `SELECT ... FOR UPDATE SKIP LOCKED`
4. **Probing** вҖ” FFprobe extracts duration, resolution, codec info
5. **Transcoding** вҖ” FFmpeg processes each resolution defined in the preset
6. **Progress** вҖ” Real-time progress parsing from FFmpeg's stderr (`time=HH:MM:SS.ms`)
7. **Completion** вҖ” Output files stored, master playlist generated
8. **Post-processing** вҖ” Thumbnail candidates, preview GIF, and banner image generated
9. **Webhooks** вҖ” `video.ready` event fired to all subscribed endpoints

### Output Structure

```

  {hash}_{videoID}/
    720p/
      playlist.m3u8
      segment000.ts
      segment001.ts
      ...
    1080p/
      playlist.m3u8

      ...

    meta.json            вҶҗ Generation metadata
thumbnails/
  {videoID}/

    ...
```

## Frontend Architecture

The Next.js frontend uses the **App Router** with a provider-heavy architecture:

```
QueryProvider
  в””в”Җв”Җ AuthProvider
       в””в”Җв”Җ ThemeProvider (next-themes)
            в””в”Җв”Җ SettingsProvider (localStorage)
                 в””в”Җв”Җ ThemeColorApplier (DB settings вҶ’ CSS vars)
                      в””в”Җв”Җ TooltipProvider
                           в””в”Җв”Җ Page Content
```

### State Management

| Type                | Technology                   | Use Case                             |
| ------------------- | ---------------------------- | ------------------------------------ |
| **Server state**    | TanStack React Query         | API data, caching, polling           |
| **Auth state**      | React Context                | User, tokens, login/logout           |
| **UI settings**     | React Context + localStorage | Sidebar, theme, locale               |
| **System settings** | TanStack Query + DB          | Branding, rate limits, player config |
| **Form state**      | React Hook Form + Zod        | Validation, submission               |

### Route Groups

| Group     | Path Pattern                   | Access        |
| --------- | ------------------------------ | ------------- |
| Landing   | `/`                            | Public        |
| Auth      | `/signin`, `/signup`, etc.     | Public        |
| Protected | `/analytics`, `/library`, etc. | Authenticated |
| Admin     | `/settings`, `/users`          | Admin role    |

## Storage Architecture

GoMPP abstracts file storage behind the `Backend` interface:

```go
type Backend interface {
    Put(path string, reader io.Reader) error
    Get(path string) (io.ReadCloser, error)
    Delete(path string) error
    Exists(path string) (bool, error)
    Size(path string) (int64, error)
    DirSize(path string) (int64, error)
    URL(path string) string
    List(prefix string) ([]string, error)
}
```

| Backend   | Config Value                  | Use Case                                  |
| --------- | ----------------------------- | ----------------------------------------- |
| **Local** | `GOMPP_STORAGE_BACKEND=local` | Development, single-server deployments    |
| **S3**    | `GOMPP_STORAGE_BACKEND=s3`    | Production, multi-server, CDN integration |

The `LocalPathResolver` interface allows FFmpeg to access files directly on disk without downloading, while S3 backends transparently download source files to a temp directory for processing.

## Webhook System

Webhooks use HMAC-SHA256 signed HTTP callbacks:

```
Event Fired
  в”Ӯ
  в”ңв”Җв”Җ Query active webhooks subscribed to event
  в”Ӯ
  в”ңв”Җв”Җ For each webhook (async goroutine):
  в”Ӯ     в”Ӯ
  в”Ӯ     в”ңв”Җв”Җ Marshal payload as JSON
  в”Ӯ     в”ңв”Җв”Җ Sign: HMAC-SHA256(webhook.secret, body)
  в”Ӯ     в”ңв”Җв”Җ POST to webhook.url with headers:
  в”Ӯ     в”Ӯ     Content-Type: application/json
  в”Ӯ     в”Ӯ     X-GoMPP-Signature: hex(signature)
  в”Ӯ     в”Ӯ     X-GoMPP-Event: event_type
  в”Ӯ     в”Ӯ
  в”Ӯ     в”ңв”Җв”Җ On failure: retry 3x with exponential backoff
  в”Ӯ     в”Ӯ     (1s, 2s, 4s)
  в”Ӯ     в”Ӯ
  в”Ӯ     в””в”Җв”Җ Record delivery in webhook_deliveries table
```

### Event Types

| Event               | Trigger                        |
| ------------------- | ------------------------------ |
| `video.uploaded`    | Video file successfully stored |
| `video.ready`       | All transcoding jobs completed |
| `video.deleted`     | Video soft-deleted             |
| `transcode.started` | Worker begins processing a job |
| `transcode.failed`  | Transcoding job failed         |
