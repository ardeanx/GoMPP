GoMPP ships with a production-ready Docker Compose stack that includes the API server, web frontend, Nginx reverse proxy, PostgreSQL, Prometheus, and Grafana.

---

## Quick Start

```bash
# 1. Clone the repository
git clone https://github.com/ardeanx/gompp.git
cd gompp

# 2. Create environment file
cp .env.example .env
# Edit .env — at minimum set GOMPP_JWT_SECRET

# 3. Start all services
cd deployments
docker compose up -d
```

The application will be available at:

| Service                 | URL                                     |
| ----------------------- | --------------------------------------- |
| Application (via Nginx) | `http://localhost`                      |
| API direct              | `http://localhost:8080`                 |
| Web frontend direct     | `http://localhost:3000`                 |
| Prometheus              | `http://localhost:9090`                 |
| Grafana                 | `http://localhost:3001` (admin / gompp) |

## Services

### gompp-api

| Property   | Value                                                 |
| ---------- | ----------------------------------------------------- |
| Base image | `golang:1.25-alpine` (build), `alpine:3.20` (runtime) |
| Port       | 8080                                                  |
| Binary     | `/bin/gompp`                                          |
| FFmpeg     | Installed via `apk` (included in runtime image)       |
| User       | `gompp` (non-root)                                    |
| Volumes    | `gompp-media:/data/media`, `gompp-temp:/data/temp`    |

The API Dockerfile uses a **multi-stage build**:

1. **Builder stage** — Compiles the Go binary with `CGO_ENABLED=0` and stripped symbols (`-ldflags="-s -w"`)
2. **Runtime stage** — Minimal Alpine with `ca-certificates`, `tzdata`, and `ffmpeg`

A non-root `gompp` user is created and given ownership of `/data` directories.

### gompp-web

| Property    | Value                                                   |
| ----------- | ------------------------------------------------------- |
| Base image  | `oven/bun:1-alpine` (build), `node:22-alpine` (runtime) |
| Port        | 3000                                                    |
| Entry point | `node server.js` (Next.js standalone output)            |

The web Dockerfile also uses a multi-stage build:

1. **Builder stage** — Installs dependencies with `bun install --frozen-lockfile`, builds with `bun run build`
2. **Runtime stage** — Copies the standalone Next.js output and serves with Node.js

### gompp-nginx

| Property   | Value                          |
| ---------- | ------------------------------ |
| Image      | `nginx:1.27-alpine`            |
| Port       | 80                             |
| Max upload | 10 GB (`client_max_body_size`) |

Nginx routes:

| Path                | Upstream         | Notes                               |
| ------------------- | ---------------- | ----------------------------------- |
| `/api/*`            | `gompp-api:8080` | API routes                          |
| `/health`           | `gompp-api:8080` | Health check                        |
| `/internal/media/*` | Filesystem       | Internal redirect to `/data/media/` |
| `/*`                | `gompp-web:3000` | All other routes → Next.js          |

All proxied requests include `X-Real-IP`, `X-Forwarded-For`, and `X-Forwarded-Proto` headers.

### gompp-db

| Property            | Value                                    |
| ------------------- | ---------------------------------------- |
| Image               | `postgres:16-alpine`                     |
| Port                | 5432                                     |
| Default credentials | `gompp` / `gompp` / `gompp`              |
| Health check        | `pg_isready -U gompp` every 5 s          |
| Volume              | `gompp-db-data:/var/lib/postgresql/data` |

### Prometheus

| Property  | Value                                   |
| --------- | --------------------------------------- |
| Image     | `prom/prometheus:v2.53.0`               |
| Port      | 9090                                    |
| Retention | 30 days                                 |
| Config    | `deployments/prometheus/prometheus.yml` |
| Alerts    | `deployments/prometheus/alerts.yml`     |

### Grafana

| Property      | Value                               |
| ------------- | ----------------------------------- |

## Volumes

| Volume            | Purpose                                     |
| ----------------- | ------------------------------------------- |
| `gompp-db-data`   | PostgreSQL data directory                   |
| `gompp-media`     | Uploaded source files and transcoded output |
| `gompp-temp`      | Temporary transcoding workspace             |
| `prometheus-data` | Prometheus TSDB data                        |
| `grafana-data`    | Grafana plugins, sessions, and state        |

<Callout type="warn">
  The `gompp-media` volume is mounted read-only in the Nginx container for
  direct media serving. Only the API container has write access.
</Callout>

## Environment Configuration

Create a `.env` file in the project root (not inside `deployments/`):

```bash
cp .env.example .env
```

The `docker-compose.yml` references `../.env` (parent directory). Docker-specific overrides are set in the `environment` section:

```yaml
environment:
  GOMPP_DATABASE_HOST: gompp-db # Docker network hostname
  GOMPP_STORAGE_BASEPATH: /data/media # Container path
  GOMPP_TRANSCODE_TEMPDIR: /data/temp # Container path
```

These override the corresponding values in `.env`.

<Callout type="error">
  You **must** set `GOMPP_JWT_SECRET` in your `.env` file. The server will not
  start without it. Generate one with: `openssl rand -hex 32`
</Callout>

## Updating

To rebuild and restart after code changes:

```bash
cd deployments

# Rebuild specific services
docker compose up -d --build gompp-api gompp-web

# Or rebuild everything
docker compose up -d --build
```

To apply database migrations, simply restart the API container — migrations run automatically on startup:

```bash
docker compose restart gompp-api
```

## Scaling Workers

The number of transcoding workers is set via `GOMPP_TRANSCODE_WORKERS` (default: 2). To scale:

```bash
# In .env
GOMPP_TRANSCODE_WORKERS=4
```

Then restart the API container:

```bash
docker compose restart gompp-api
```

<Callout type="info">
  Each worker can run one FFmpeg process at a time. Set workers based on
  available CPU cores and memory. A good starting point is `cores / 2`.
</Callout>

## Backup & Restore

GoMPP includes backup and restore scripts:

### Backup

```bash
./scripts/backup.sh
```

This creates a timestamped backup of the PostgreSQL database and media files.

### Restore

```bash
./scripts/restore.sh <backup-file>
```

## Production Checklist

Before deploying to production:

- [ ] Set a strong `GOMPP_JWT_SECRET` (32+ random bytes)
- [ ] Change PostgreSQL credentials (`POSTGRES_USER`, `POSTGRES_PASSWORD`)
- [ ] Change Grafana admin password (`GF_SECURITY_ADMIN_PASSWORD`)
- [ ] Configure `GOMPP_SERVER_CORS_ORIGINS` with your domain
- [ ] Set `GOMPP_WEBAUTHN_RP_ID` and `GOMPP_WEBAUTHN_RP_ORIGINS` to your domain
- [ ] Configure TLS termination (add SSL to Nginx or use a load balancer)
- [ ] Set appropriate rate limits (`GOMPP_RATELIMIT_*`)
- [ ] Configure S3 storage for production workloads (or ensure local volumes have sufficient disk)
- [ ] Set `GOMPP_LOG_FORMAT=json` for structured logging
- [ ] Configure Prometheus alerting rules
- [ ] Set up regular database backups
- [ ] Review `client_max_body_size` in Nginx config for your upload requirements

### TLS Configuration

For HTTPS, update the Nginx config to include SSL:

```nginx
server {
    listen 443 ssl http2;
    server_name yourdomain.com;

    ssl_certificate     /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;

    # ... existing proxy rules ...
}

server {
    listen 80;
    server_name yourdomain.com;
    return 301 https://$host$request_uri;
}
```

Mount your certificates as a volume in the Nginx container.
